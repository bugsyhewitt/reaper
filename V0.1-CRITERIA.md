## Cut line (one sentence)

reaper v0.1 brings the **HTTP/2 single-packet attack** to a headless Python 3 CLI — benchmark-then-burst race detection with statistical deviation confirmation and an evidence report — where PortSwigger's Turbo Intruder is Burp/Jython-locked. Without the single-packet attack, reaper is just `race-the-web` again; **it is the entire differentiator and is mandatory for v0.1.**

## v0.1 acceptance criteria (testable)

1. **HTTP/2 single-packet attack (MANDATORY differentiator).** Built on `hyper-h2` (sans-IO) + raw `socket`/`ssl` — no scapy, no root, pure Python 3 for the 20–30 request case. Multiplex N requests on one HTTP/2 connection; **withhold each request's final frame** (empty DATA with END_STREAM for bodyless; the last body byte otherwise); after a ~100ms settle, **release all withheld frames in a single flush** so they land in one TCP packet. Control `TCP_NODELAY` (disable to batch via Nagle) and optionally warm with a PING. Target **20–30 concurrent requests**. Include **h2c cleartext** (needed for CI and some targets).
2. **HTTP/1.1 last-byte-sync fallback + connection warming.** Auto-selected when the target is HTTP/1.1-only or refuses enough concurrent H2 streams. One TCP connection per request, withhold the final byte, flush all final bytes together; prime connections first to clear TCP slow-start.
3. **Single-endpoint limit-overrun scenario** (N identical requests, one synchronized gate) — the 80% case (over-redeem coupon, over-withdraw balance).
4. **Minimal multi-endpoint mode.** A heterogeneous request group (different methods/paths/bodies) sharing a session/auth context, with **manual** per-request ordering/delay (NOT auto-calibrated) and a single synchronized release. Included because MFA/OTP and email-confirm sub-state races are core differentiators over race-the-web.
5. **Benchmark + statistical deviation confirmation.** Send the sequential baseline first (record status codes, response-body content/hash, timing, second-order effects); then the concurrent burst; then **flag deviations** (status / body-hash / timing / second-order signal). Emit an **evidence report**: baseline-vs-burst diff, count of anomalous responses (e.g. "2 of 20 returned 200 where baseline gave one 200 + rest 409"), timing distribution, reproducibility. **Guard the final-state false positive** — an early race win that later requests overwrite/invalidate is not a real finding.
6. **Architecture (mandatory split).** The suite HTTP client **cannot** drive the synchronized burst (single-packet needs raw frame/socket control). Use the shared client for **auth/session setup + the sequential baseline**; a **dedicated low-level h2 engine** drives the burst.
7. **Output.** Confirmed races emitted in the suite finding schema + SARIF 2.1.0 + HackerOne markdown via `h1-reporter`.
8. **R5 (untrusted input):** treat all target response bytes as **data** during deviation analysis — no `eval`, no LLM-judging of response content.
9. **CI test lab** (see Testability) + the suite `ship_gate` wheel test.

## Explicitly NOT in v0.1 (deferred)

- **First-sequence-sync / >65KB bodies / >~30 requests** (RyotaK: IP fragmentation + TCP sequence reordering at OSI L3–L4; needs **scapy, raw sockets, root**) → **v0.2.** Could consume the whole 500K budget on socket-portability + privilege edge cases; the 20–30 request ceiling covers nearly all limit-overrun work (OTP-brute-scale races are the rare exception).
- **HTTP/3 single-datagram attack** (QUICker territory) — new protocol surface, low prevalence.
- **Auto-calibration of multi-endpoint delays** (Kettle's client-side timing) — hard; ship manual delays first.
- **Endpoint auto-discovery, distributed/multi-host bursts, SOCKS/proxy chaining, any GUI.**

## Testability (CI-friendly lab)

Build a **deliberately race-vulnerable local app** as a CI fixture with a **non-atomic check-then-act** and a **small artificial delay to widen the window**:

- **Fixture A — single-use coupon:** `POST /redeem {code}` → check `unused`, `sleep(~0.1–0.5s)`, mark used + credit. **Fixture B — wallet overdraw.** **Fixture C — multi-endpoint sub-state** (`/email/change` + `/email/confirm` sharing one pending slot). Backing store without row locks (SQLite / in-memory dict).
- **Two make-or-break requirements:** (1) the fixture MUST run **concurrently** (Flask `threaded=True` / multi-worker gunicorn / async) — a single-worker server serializes requests and no race ever occurs (silent false-negative); (2) serve **HTTP/2 via Hypercorn (h2 + h2c cleartext, no TLS certs needed)** for the single-packet path — **uvicorn does NOT support HTTP/2.** Use any threaded HTTP/1.1 server for the last-byte-sync fixture.
- **Deterministic gate** (mirrors the tool's own methodology): **control** = N sequential single-use-coupon redemptions → **exactly 1 success** (proves the fixture is correct, not broken); **attack** = N concurrent redemptions via reaper → **>1 success** (an over-limit condition sequential requests cannot produce). Run a `delay=0` variant as a **non-gating soak test** to exercise tight windows without flaking the CI gate.

## Safety

- **R5 (untrusted input):** response bytes are **data, never instructions**.
- **Scope enforcement before ANY burst** via **`scan-primitives`** (auth/baseline requests); the low-level h2 burst engine must honor the same `Scope` object.
- Concurrency is inherently higher-impact on a target than a normal scan — respect scope + rate posture; the burst is bounded (20–30 requests) in v0.1.

## Shared infra (resolved — Option C, 2026-07-02)

- **HTTP client:** use **`scan-primitives`** for auth/session setup + the sequential baseline only. reaper's synchronized burst is **raw by necessity** (a dedicated low-level `hyper-h2` engine) — the shared client cannot drive it. Pass the `Scope` object into the burst engine so it is scope-checked too.
- **Findings:** per-tool `Finding` to the **pinned contract** in `scan-primitives/SPEC.md` (`cwe_id=362`); SARIF 2.1.0; adopt **`h1-reporter`**.
- Build **`scan-primitives`** before this tool.
